# 1.检索引擎：单一路径 → 双路并发 + RRF 融合

目标： 解决原方案中结构化检索（searchTypes.map）结果集太大，BM25/HNSW 被迫拼接巨长 SQL 的问题。

变更：

- 删除 旧的结构化检索 + 变体循环 + 质量门控 + EvalTrace 逻辑，约 400 行
- 新增 runConcurrentSearch() 并发执行两路独立检索：
- BM25 路 — PostgreSQL FTS ts_rank，WHERE type=ANY(...) 单 SQL 过滤多类型
- HNSW 路 — pgvector vector_cosine_ops 近似最近邻，同样 type=ANY(...) 过滤
- 新增 fuseByRRF() — Reciprocal Rank Fusion（k=60）合并两路结果
- 修复 原逻辑中每个 type 一次 SQL 的 N+1 查询问题

涉及文件： src/services/recommendationService.ts

# 4. Embedding 空间对齐：双端 LLM Prose

目标： 解决索引侧（代码→语义指纹）和检索侧（用户 query→语义模板）embedding 分布不一致导致的召回差问题。

变更：

- 检索侧 — callLLMToRewrite()：用户 query + 关键词 → Ollama 生成英文自然语言 prose
- 索引侧 — codeToLLMProse()：代码元信息 → Ollama 生成同分布英文 prose，持久化到 content 字段
- 降级回退： 检索侧 LLM 不可用 → 关键词拼接；索引侧 LLM 不可用 → 模板 toEmbedDoc()/indexedRowToEmbedText()
- embeddingWorker.ts — reindex 处理 embedding 前先调 codeToLLMProse，将 prose 写入 content 后再 embedding，确保两端的向量空间一致

涉及文件： src/services/queryRewriter.ts（新文件）, src/workers/embeddingWorker.ts

# 5. LLM 结果缓存：内存 Map → Redis

目标： 解决 100 人团队场景下内存 Map 持续膨胀 + 进程重启缓存丢失的问题。

变更：

- 删除 REWRITE_CACHE Map（500 条上限 + O(n) 逐出）
- 删除 CODE_PROSE_CACHE Map（2000 条上限 + O(n) 逐出）
- 改用统一 Redis 缓存（与 BullMQ 共用同一 env.redisUrl）：
    - rewrite:{sha256} → TTL 1 小时（query rewrite 结果）
    - codeprose:{semanticHash} → TTL 24 小时（代码 prose 结果）
- 懒初始化 getRedis()，Redis 不可用时静默降级为无缓存
- 导出 closeRedis() 供进程优雅关闭

涉及文件： src/services/queryRewriter.ts
